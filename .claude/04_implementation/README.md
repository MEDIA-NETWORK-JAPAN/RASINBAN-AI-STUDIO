## 🎯 使い方

### 1. 実装開始前

**Step 1:** マスタープランを読む
```bash
cat .claude/04_implementation/00_実装計画マスタープラン.md
```

**Step 2:** 現在のPhaseの「実装対象」「参照ドキュメント」「成果物」を確認

**Step 3:** 依存関係を確認（前のPhaseが完了しているか）

### 2. 実装中

**Step 1:** 対象コンポーネント/画面の仕様書を読む
- UI設計: `.claude/02_ui_design/screens/` または `.claude/02_ui_design/components/`
- テストケース: `.claude/01_development_docs/03_テストケース定義書.md` または `04_画面テストケース定義書.md`

**Step 2:** 実装

**Step 3:** テスト実行
```bash
vendor/bin/sail artisan test --compact --filter=TestName
```

**Step 4:** コードフォーマット
```bash
vendor/bin/sail bin pint
```

### 3. 実装完了後

**Step 1:** レビュー依頼（ユーザーに報告）

**Step 2:** レビュー承認後、コミット

**Step 3:** マスタープランの進捗テーブルを更新
- ステータス: 🔲 → ✅
- 完了日を記入

## ❓ FAQ

### Q1: Phase の順序を変更できますか？

**A:** 依存関係があるため、基本的には順序を守ってください。特に：
- Phase 0（基本UIコンポーネント）は Phase 1（認証機能）の前提
- Phase 1（認証機能）は全画面実装の前提
- コンポーネント実装は対応する画面実装の前提

### Q2: Phase内の実装順序は自由ですか？

**A:** Phase内のコンポーネント/画面は並行実装可能です。ただし、テストケース定義書を確認し、依存関係がないことを確認してください。

### Q3: テストが失敗した場合は？

**A:** 以下の手順で対応：
1. エラーメッセージを確認
2. テストケース定義書の「期待値」を再確認
3. 実装を修正
4. 再テスト
5. それでも解決しない場合、ユーザーに質問

### Q4: markTestIncomplete と markTestSkipped の違いは？

**A:**
- **markTestIncomplete:** 実装予定だが未実装（実装後に有効化）
- **markTestSkipped:** Alpine.jsなどJavaScript動作のテストで、PHPUnitでは困難なため一時的にスキップ

### Q5: レビュー前にコミットしてもいいですか？

**A:** いいえ。ユーザーの指示により「今後は修正後、レビューした後にコミットして」という運用ルールになっています。必ずレビュー承認後にコミットしてください。
